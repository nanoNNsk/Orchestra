🎻 Orchestra

Chat with an AI company, not a chatbot.

A team of specialized AI agents that works together like a real company. You talk to one person — the CEO — and the rest of the organization handles the work behind the scenes.

Chairman (you) ⇄ CEO ▶ the right department

You set the direction. The CEO delegates, collects results, and reports back one consolidated summary — never the internal chatter.

01Core concepts

TermWhat it is
ChairmanYou. The human. The only one the CEO talks to.
CEOThe single entry point. Plans, routes to departments, reports back.
DepartmentA stable bucket of related expertise. Has one head. Created by the CEO.
HeadLeads a department. Routes work to its roles — and hires new ones.
RoleA specialist = system prompt + skills. Persistent. The unit that gets hired.
SkillAn atomic capability (web search, write code…). Created by humans only.
Two things both called “role”
Role definition
prompt + skills. Tiny text. Permanent. This is what gets hired.
Role memory
context it builds doing work. Grows. Loaded only when active.

500 definitions are cheap — only memory costs, and only for active roles. That’s why the roster can grow without bound.

02How a request flows

  1. Chairman → CEO. You send a message. The CEO is the only entry point.
  2. CEO routes to a department. Picks the one that fits (creates a new one if none do).
  3. Head routes to a role. Reuses a fitting role — or hires a new one.
  4. Role executes using its skills, and accumulates memory for next time.
  5. Results bubble up: head reviews → CEO consolidates → you get one summary.

03The hiring model

Coverage grows on demand without bloat. Who can create what:

ActorCan create
Chairman / devsSkills — real capabilities, never self-minted by agents
CEODepartments
Department headRoles inside its own department
Hire protocol (run by a department head)
SCAN→ MATCH→ HIRE→ APPEND→ LOG

Read each role’s one-line description → can someone do this? If yes, reuse. If no, write a prompt + pick existing skills, append to the roster, and tell the Chairman. No automatic firing — crowded rosters get a manual review.

05Architecture & stack

A two-layer hierarchy over the Codex SDK — the mapping is 1:1, so almost no orchestration is hand-written:

OrchestraMaps to (Codex SDK)
CEOThe top-level agent loop
Head & roleA Codex agent instance (system prompt + tool allowlist)
SkillA tool the agent may call
Role definitionData in orchestra.json
Role memoryA per-role memory file

Stack:

TypeScript · Node ≥ 20 Codex SDK OpenAI models — stronger for CEO/heads, faster for workers Flat-file storage (JSON + memory) React + Vite chat UI validate.py

Deliberately skipped (until something forces it):

no database no vector search for hiring no auth / multi-tenant no message queue no auto-firing / TTL no frontend state library

06Why hierarchy beats a flat roster

The CEO chooses among ~20 departments, not ~150 roles. Routing is O(departments) then O(roles in one department) — never O(everyone). That is the whole reason coverage can grow without the routing problem blowing up. Definitions are data, not code, so the system can hire into itself at runtime by appending JSON.

Orchestra · early stage — roster, hiring protocol & validator are runnable today; the agent runtime is next.
